<grid_architecture_decision>
CHOSEN: Architecture (a) — runtime-measured bound. Justification with evidence:

- (b) "drop the inner scroller, use main page scroll" is REJECTED because the inner scroller is load-bearing for THREE coupled behaviors that (b) would regress:
  1. Per-column STICKY HEADER band `sticky top-0 z-10` (category-column.tsx:171) + top backdrop mask `sticky top-0` z-5 (category-column.tsx:147) pin against the INNER scroll container's top. Re-anchored to the main page scroll context they would pin at viewport top — UNDER the pinned `[data-shell-header]` (sticky top:0) and the `[data-bdp-tabs]` band (top: calc(4rem+1px), global.css:501-503) — i.e. hidden behind chrome. Matching them to the band would require a per-column `top` offset tracking the DYNAMIC band height (header inset + banner) — exactly the brittle math (a) eliminates.
  2. Horizontal column scroll (`overflow-auto`, both axes) — the grid is `flex gap-2 w-fit` wider than viewport (spendings-grid-client.tsx:445,452).
  3. The custom pull-to-refresh suppressor + `gridScrolled` shadow cue read `gridRef.current.scrollTop` (spendings-grid-client.tsx:209-283). On main page scroll these break.
     The bdp-tab-frame.feature contract even documents "Spendings has an inner scroll container with overscroll-behavior:contain" as the reason it is exempt from the rubber-band guard (lines 38-42). Dropping it would also break that contract.

- (c) "fix constants" is FORBIDDEN by the constraints — two device failures prove constants rot.

- (a) keeps ALL existing couplings intact and fixes the ROOT bug: viewport math cannot know the element's top offset. A ResizeObserver measures `scroller.getBoundingClientRect().top` against `visualViewport.height` and writes the resolved height into a CSS var — self-correcting on every band/inset/keyboard change. The in-flow tail spacer is retained (iOS WebKit end-of-scroll truth, SHELL-R8..R10) and made mode-aware (browser: bar clearance; standalone: env+64px).
  </grid_architecture_decision>
