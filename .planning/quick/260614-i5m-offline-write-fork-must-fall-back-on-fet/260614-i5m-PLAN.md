<objective>
Bug fix (Phase 08 UAT test 4, iOS device-confirmed): offline quick-entry shows a
perpetual LOADING SPINNER instead of the Clock + "Pending" offline marker.

Root cause (verified against current code 2026-06-14):

1. `use-create-transaction.ts:93` forks to the offline queue ONLY when
   `navigator.onLine === false`. iOS Safari/PWA reports `onLine === true` (stale)
   with a dead network → the write falls through to the real POST.
2. The POST at `:113` has NO timeout/abort → it hangs forever.
3. `pending:true` is set in onMutate (`:145`); spinner = `Loader2 animate-spin`
   gated on `txn.pending` (transaction-row.tsx:446). `pending` only clears in
   onError (`:173`) / onSuccess (`:188`) — neither fires on a hung fetch → stuck.
4. Clock + "Pending" marker is gated on `isOfflinePending` (row's key present in
   IndexedDB queue, transaction-row.tsx:480). The queue insert runs ONLY in the
   skipped offline fork → never written on the online path → marker never shows.

Purpose: make the offline fallback FETCH-FAILURE-driven (not solely
navigator.onLine), with a write timeout, so a dead-network write on iOS enqueues,
clears the spinner, and shows the Clock + "Pending" marker.

Output: spinner clears + Clock/"Pending" marker shows when the network is dead
even though navigator reports online; idempotency preserved (no double-write).
</objective>

    ```ts
    // Build the payload once (shared by POST and the fallback enqueue).
    const payload = {
      date: input.date,
      category_id: input.categoryId,
      amount_original_cents: input.amountCents,
      currency_original: input.currency,
      note: input.note ?? null,
    };

    // Helper: enqueue under the SAME key, then route to onError (keep the row).
    const fallbackToQueue = async () => {
      await enqueueOfflineTxn({
        idempotencyKey: key,
        budgetId,
        payload,
        enqueuedAt: new Date().toISOString(),
      });
      throw new OfflineEnqueuedError();
    };

    let res: Response;
    try {
      res = await clientApiFetch(`/budgets/${budgetId}/transactions`, {
        method: "POST",
        headers: { "Content-Type": "application/json", "Idempotency-Key": key },
        body: JSON.stringify(payload),
        // Write-only timeout: a dead network (iOS reports onLine=true) rejects
        // fast with AbortError instead of hanging the optimistic row forever.
        // Scoped to THIS POST — reads via clientApiFetch are untouched.
        signal: AbortSignal.timeout(8000),
      });
    } catch {
      // Network throw (TypeError "Failed to fetch") or AbortError (timeout) —
      // the server was unreachable. Enqueue for later replay.
      return await fallbackToQueue();
    }

    if (!res.ok) {
      // Server-UNREACHABLE-class status (5xx) → enqueue + retry later.
      if (res.status >= 500) return await fallbackToQueue();
      // Genuine client error (4xx) → real error, do NOT enqueue (would loop).
      throw new Error(await res.text());
    }
    return (await res.json()).transaction;
    ```

    Notes:
    - `return await fallbackToQueue()` — fallbackToQueue always throws, the
      `return` just satisfies the type checker / control flow; OfflineEnqueuedError
      propagates to onError which keeps the row (pending:false, unsent:true).
    - DO NOT touch onMutate / onError / onSuccess / onSettled — onError already
      sets pending:false + unsent:true, and the queued key (same `key`) drives the
      isOfflinePending Clock marker via OFFLINE_QUEUE_CHANGED_EVENT.
    - DO NOT add a timeout to clientApiFetch / budget-fetch.ts (reads unchanged).
    - AbortSignal.timeout is available in happy-dom + modern Safari; if the test
      env lacks it, polyfill in the test, not in production code.
