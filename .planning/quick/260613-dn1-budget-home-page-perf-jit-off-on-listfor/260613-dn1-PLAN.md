<chosen_pool_max>
PG `max_connections = 100` (infra/postgres/postgresql.conf:14), no jit override.

Per-process pool tally:

- api process: ONE appPool singleton (module-level \_appPool). With `max: 25` → up to 25 conns.
- worker process: separate workerPool (DATABASE_URL_WORKER) — untouched, pg default 10.
- migrator: short-lived migratorPool — default 10, only during migrate.
- Postgres reserves ~3 superuser_reserved_connections by default.

Worst-case steady state: api 25 + worker 10 = 35, well under 100 even with a second api replica (50) + migrator burst (10) + superuser (3) = 63. `max: 25` chosen over 20-30 to comfortably absorb the ~72 concurrent-tx burst (6 tx/card x 12 cards) the home load triggers without starving — pg queues beyond max rather than erroring, so 25 turns the 6.7x serial collapse into near-parallel while staying safely inside headroom. Only appPool changes.
</chosen_pool_max>

<jit_fix_mechanism>
Exact injection point — `packages/tenancy/src/adapters/persistence/workspace-repo.ts`, inside `listForUser`'s `withUserContext` callback:

- AFTER the existing member-budget-ids `SET LOCAL app.tenant_ids` block (current line ~99).

Why it is correct + safe:
