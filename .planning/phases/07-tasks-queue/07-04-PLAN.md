<objective>
Wire the CONFIRM_DRAFT generator (emit on draft INSERT) and three auto-resolve
paths (confirm, dismiss, skip). Smallest of the three generators because the
math is "row exists ⇒ task exists; row resolved ⇒ task resolved" — no FX, no
multi-mutation surface area.

Per D-PH7-08 (emission point), D-PH7-09 (resolve on confirm), D-PH7-10 (resolve
on dismiss + skip), D-PH7-11 (dedup via partial unique index on
payload_json->>'draft_id'), D-PH7-12 (payload shape). Pitfall 3 (RESEARCH.md):
emission gate MUST be `insertResult.rows.length > 0` — `ON CONFLICT DO NOTHING`
returns 0 rows when the row pre-existed.

Purpose: Recurring drafts are the highest-frequency task source. Every materialized
draft becomes a row that needs user confirmation. Without auto-resolve on confirm,
the banner would grow forever even after the user took action.

Output: 4 modified files + confirm-draft.test.ts converted from scaffold to
6 passing Nyquist tests.
</objective>

| ---------- | ---------------------- | --------------------------------------------------------- | ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-07-04-01 | Tampering              | Draft INSERT atomicity with task emit                     | mitigate    | Both writes in same `withTenantTx` — Postgres tx atomicity guarantees both succeed or both rollback. Verified by integration test #1 (fresh INSERT produces task; failed INSERT produces no task).               |
| T-07-04-02 | DoS                    | Concurrent recurring-engine runs spam CONFIRM_DRAFT tasks | mitigate    | Partial unique index on `(payload_json->>'draft_id') WHERE kind='CONFIRM_DRAFT' AND status='PENDING'` enforces at-most-one PENDING task per draft. ON CONFLICT DO NOTHING absorbs the race. Verified by test #2. |
| T-07-04-03 | Information Disclosure | rule_name payload leaks across tenants                    | accept      | Payload contains data from the rule which is already tenant-scoped via RLS at fetch time. Task row's tenant_id matches; RLS on tasks_tenant_isolation prevents cross-tenant read.                                |
