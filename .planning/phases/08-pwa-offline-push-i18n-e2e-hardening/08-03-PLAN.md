<objective>
Offline cache + write-replay + cache hydration on fetch. Build the idb cache layer (budgets/wallets/categories/transactions, all visited months — D-01), the offline write queue, the `use-online-sync` reconnect-replay hook (best-effort, same-key idempotency — D-02), the offline fork in `use-create-transaction` that enqueues when `!navigator.onLine`, the WRITE-PATH wiring that fills the cache on every successful fetch (so offline reads actually hit — B4), and the two offline UI surfaces (global queue badge + sync-issues list). The cache wipe helper enforces tenant isolation on logout/switch (D-06).

Purpose: This is the offline half of the phase, independent of the push pipeline (RESEARCH critical-order Step 3 is parallel to Steps 1–2). The cache is only useful if something writes to it — the fetch-success wiring populates budgets/wallets/categories/current-month transactions and records the last-synced timestamp that feeds the D-05 staleness marker. The server-side idempotency middleware already dedupes — the client only has to re-use the same UUID on replay.
Output: cache/queue/sync/cache-on-fetch modules + offline fork + 2 components + 4 green Vitest suites (turning the 08-01 scaffolds green).
</objective>

## Trust Boundaries

| Boundary                   | Description                                                              |
| -------------------------- | ------------------------------------------------------------------------ |
| IndexedDB ↔ origin         | Per-origin readable store — must hold no secrets, no other tenant's data |
| offline queue → API replay | Replayed write must not double-commit                                    |

## STRIDE Threat Register

| Threat ID  | Category        | Component                 | Disposition | Mitigation Plan                                                                                                                                                     |
| ---------- | --------------- | ------------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-08-03-01 | Info Disclosure | offline-cache (IndexedDB) | mitigate    | Cache holds only budget domain data (no auth tokens, no secrets); wipeBudgetCache() on logout/tenant-switch; cross-tenant-cache.spec.ts guards the wipe (ASVS V3)   |
| T-08-03-02 | Tampering       | replay double-write       | mitigate    | Replay re-uses the original Idempotency-Key; server middleware (scope_hash+body_hash+SELECT FOR UPDATE) returns the cached 2xx — no second row (RESEARCH Pattern 2) |
| T-08-03-03 | DoS             | unbounded queue growth    | accept      | D-06: no hard cap in v1.1; queue drains on reconnect; revisit only if growth becomes a real problem                                                                 |

</threat_model>
