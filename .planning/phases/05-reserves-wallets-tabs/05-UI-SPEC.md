# Phase 5 — UI Design Contract: Reserves & Wallets Tabs

> Visual and interaction contract for the two remaining BDP tabs (Reserves + Wallets). Generated by gsd-ui-researcher; verified by gsd-ui-checker.
>
> **Source-of-truth precedence:** `DESIGN.md` > `apps/web/src/app/global.css` > Phase 3 UI-SPEC carry-forward > Phase 4 UI-SPEC carry-forward > this document. Every token, color, font, and spacing value below is **already defined upstream** — this spec only describes how they compose into the new surfaces.
>
> **Carry-forward locked (do NOT redefine):** dark-only theme, 4-multiple spacing scale, Inter (`--font-sans`) + IBM Plex Sans (`--font-numeric`) stacks, single-yellow-accent scarcity rule, `--info-ring` focus, BDP sticky pill tabs at `top: 64px`, task banner above tabs, `<Sheet>` primitive available, `sonner` toast primitive available, `<AlertDialog>` primitive available, `@dnd-kit` sensor calibration (Phase 4 spendings-grid).
>
> **Deliberate divergence from Phase 4:** Wallets tab explicitly opts OUT of the spendings-grid `D-PH4-INT1` no-hover universal rule. Reasoning: wallets are form-like, sparse, edited rarely but directly. See §Interaction Model.

| Role             | Token                         | Size | Weight | Line Height | Font stack                                                             | Phase 5 usage                                                                                                                                                                                                                                |

| Title            | `title-sm` (`.text-title-sm`) | 16px | 600    | 1.4         | Inter                                                                  | AlertDialog header ("Delete wallet 'Groceries'?"), `<MismatchChip>` direction label ("Overfunded" / "Underfunded" / "Reconciled"), `Σ` totals row labels                                                                                     |
| Numeric (medium) | `num-md` (`.text-num-md`)     | 16px | 500    | 1.4         | IBM Plex Sans (`--font-numeric`), `font-variant-numeric: tabular-nums` | All currency values: wallet amount (resting + editor), reserve balance (resting + editor), wallet share amount, totals `Σ category reserves` / `Σ reserve wallets`, mismatch chip amount                                                     |
| Numeric (small)  | `num-sm` (`.text-num-sm`)     | 14px | 500    | 1.4         | IBM Plex Sans                                                          | Wallet share percentage (the `(XX%)` parenthetical next to the share amount); secondary numeric annotations in totals row                                                                                                                    |

**Section header treatment** (Wallets sections + Reserves Active/Excluded):

- Class: `.text-caption` (`12px / 500 / Inter`)
- Letter-spacing: `0.06em` (uppercase tracking — applied inline via Tailwind `tracking-wider`)
- Text-transform: `uppercase`
- Color: `--muted-foreground` (`#707a8a`)
- Examples: `SPENDINGS WALLETS` · `CUSHION WALLETS` · `RESERVE WALLETS` · `ACTIVE` · `EXCLUDED`

**Reserves table column headers:**

- Same `.text-caption` + uppercase tracking treatment
- Color: `--muted-foreground`
- Examples: `CATEGORY` · `RESERVE BALANCE` · `WALLET SHARE` · `ACTIONS`

**Weights declared: exactly 2 — 400 (body) and 600 (title).** Numeric role uses weight 500 per DESIGN.md `number-md`/`number-sm` tokens — single tabular-numerals exception that is NOT a new editorial weight (ships in `global.css` already, identical to Phase 3+4 carry-forward).

**Body line-height:** 1.5 (DESIGN.md `body-md`). **Title line-height:** 1.4 (DESIGN.md `title-sm`). **Caption line-height:** 1.4.

**Don't:**

- Don't render any currency value in Inter — every monetary number routes through `.text-num-md` or `.text-num-sm` (IBM Plex Sans + tabular-nums) per DESIGN.md L594.
- Don't introduce `display-*` sizes — both tabs are dense product surfaces, not marketing.
- Don't soften title weight to 400 — DESIGN.md L604.
- Don't bold wallet names or section headers below 600 — body weight 400 for names, 500 for section captions.
- Don't render em-dash placeholders (`—`) in Inter — they go inside `.text-num-md` cells so they vertically align with surrounding numeric values.

**Accent (10% yellow) reserved-for list — Phase 5 EXACT inventory. No other element may be yellow.**

1. **Sticky pill tab "Reserves" — yellow active background** (Phase 3 carry-forward — only yellow on the BDP frame above the tab content when the user is on this tab).
2. **Sticky pill tab "Wallets" — yellow active background** (Phase 3 carry-forward — same).
3. **AlertDialog primary action button** in `<WalletDeleteConfirm>` — wait, NO. The dialog primary is destructive (red), not yellow. **There is NO yellow inside Phase 5 tab content this phase.** Yellow appears ONLY on the BDP pill tab indicator carried over from Phase 3.

**Justification for empty yellow inventory inside tab content:** Both Reserves and Wallets are data-table surfaces with no "primary action that converts a user." Adding a wallet is a low-stakes inline operation (dashed `+ Add wallet`, NOT a yellow CTA per D-PH5-E6 — visual rhyme with Phase 4 dashed +Add column which is also not yellow). Mismatch chip is informational (Phase 7 owns explicit task-driven CTAs). Reserves Actions column is inert this phase. The scarcity rule (DESIGN.md L592) holds: yellow stays reserved for tab indicator + Phase 7 task-banner primary actions.

**Destructive (red) reserved-for list — Phase 5 EXACT inventory:**

1. **Trash icon stroke** in `<WalletRow>` revealed actions — stroke `--destructive` resting. (Confirmation dialog before delete fires.)
2. **Delete button inside `<WalletDeleteConfirm>` (AlertDialog)** — `Button variant="destructive"` from `ui/button.tsx`. Black text on red background per shadcn convention.
3. **Mismatch chip variant `overfunded`** — chip background transparent, 1px border `--destructive`, leading icon + amount in `--destructive` (`#f6465d`). Per DESIGN.md L603, no red fill — border + text color only.
4. **Mismatch chip variant `underfunded`** — same treatment as `overfunded`. Both are "warning tone" per D-PH5-R12; we use `--destructive` because the project has no neutral-warning token and overspending vs underfunding both signal "action needed". Helper sentence body uses `--muted-foreground` (not red) so the chip doesn't shout.
5. **`<InlineEditCell>` editor 1px red ring** when a PATCH has just failed (mirrors Phase 4 quick-entry input failed-state). Ring clears on retry success.

### `<InlineEditCell>` lifecycle (shared between Wallets cells + Reserves balance cell)

1. Resting: cell renders `props.render(value)`.
2. Click (or Enter key when focused): `editing = true`; `props.renderEditor(draft, setDraft, commit)` mounts; editor auto-focuses.
3. Blur (or Tab, or Esc):
   - Esc: discard draft; `editing = false`; no mutation.
   - Tab / blur with `draft === value`: no mutation; `editing = false`.
   - Tab / blur with `draft !== value`: `await props.onSave(draft)` fires; spinner appears after 200ms threshold; on resolve toast + `editing = false`; on reject toast.error + draft reverts + 1px destructive ring persists until next interaction.

### `<DashedAddButton>` lifecycle (Wallets only)

1. Resting: button renders dashed border `--hairline-dark` (1px solid? NO — **2px dashed `--muted-foreground`** at 80% opacity), transparent background, label centered `.text-body-md` color `--muted-foreground`, leading `Plus` icon 16px.
2. Hover (desktop): border → 2px dashed `--body-on-dark`; background → `--surface-elevated-dark` at opacity 0.4; label color → `--body-on-dark`.
3. Focus (keyboard): standard `--info-ring` 2px ring (Phase 3+4 carry-forward).
4. Click / Enter:
   - Optimistic: a new blank `<WalletRow>` is inserted ABOVE the button with `name=""` (placeholder "Wallet name"), `currency = budget default`, `amount = 0`, `type = section type` (locked).
   - Name cell auto-enters edit state with focus.
   - Dashed button moves visually below the new row (always last in section).
   - On Name blur with non-empty value → optimistic POST /wallets fires; if server accepts, row gains `id` and persists; if reject, row removed + toast `wallets.toast.createFailed`.
   - On Name blur with empty value → row is silently removed (no toast). User can re-click dashed button to retry.

- Phase 7 may wire this chip to expand a task-driven primary action (or remove it entirely in favor of a banner). Phase 5 ships pure information.

## Accessibility

- **All drag handles** carry `aria-label="Drag to move {name} to another section."` and are keyboard-activatable via KeyboardSensor (Phase 4 carry-forward).
- **All `<InlineEditCell>` cells** carry `aria-label` per the table above; cell wrapper is `role="button"` + `tabindex="0"`; pressing Enter or Space enters edit state.
- **AlertDialog** uses shadcn `<AlertDialog>` which is Radix-backed (full a11y).
- **Section headers** are rendered as `<h3>` semantic HTML so screen readers announce section context when navigating into a section.
- **Mismatch chip** uses `role="status"` so screen readers re-announce when the chip variant changes (e.g. user finishes rebalancing → chip transitions from `underfunded` to `reconciled`).
- **Reserves Actions column placeholder** has `aria-hidden="true"` because it is decorative/inert this phase.
- **Em-dash placeholders** (`—`) are read by screen readers as "em dash"; for share column empty state, the cell carries `aria-label="No share — no reserves yet"` so the experience reads naturally.
- **Color is never the sole indicator** — Excluded row dimming is paired with the explicit "Excluded" section header; mismatch chip color is paired with the `+` / `−` glyph and the explicit direction word.
- **Focus order** flows: section header → first row drag handle → row cells (Name → Currency → Amount or Category → Balance) → trash icon (Wallets only) → next row → dashed `+ Add wallet` (Wallets) → next section.
