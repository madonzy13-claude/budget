# Phase 9 — UI Design Contract: Investments Wallet

> Visual and interaction contract for the Investments section — the last section of the Wallets tab on the Budget Detail Page. Generated by gsd-ui-researcher.
>
> **Source-of-truth precedence:** `DESIGN.md` > `apps/web/src/app/global.css` > Phase 5 UI-SPEC carry-forward > Phase 4 UI-SPEC carry-forward > this document. Every token, color, font, and spacing value below is already defined upstream. This spec only describes how they compose into new Phase 9 surfaces.
>
> **Carry-forward locked (do NOT redefine):** dark-only theme, 4-multiple spacing scale, Inter (`--font-sans`) + IBM Plex Sans (`--font-numeric`) stacks, single-yellow-accent scarcity rule, `--info-ring` focus ring, BDP sticky pill tabs at `top: 64px`, `<Sheet>` primitive, `sonner` toast, `<AlertDialog>` primitive, `@dnd-kit` sensor calibration (TouchSensor `{delay:300, tolerance:5}`, PointerSensor distance 4px), `<RowDragHandle>` atom, `<DashedAddButton>` atom, `clientApiWrite` optimistic mutation pattern.
>
> **Deliberate divergences from Phase 5 (documented here, not inherited):**
>
> 1. P/L coloring uses `--trading-up` / `--trading-down` as a SEMANTIC EXCEPTION to the single-yellow-accent rule — financial gain/loss is a universal data semantic, not a CTA accent. Yellow stays reserved for its existing roles. Identical to Phase 4's overspent/balance treatment.
> 2. Collapsible group headers — Phase 5 wallet sections are non-collapsible; Phase 9 groups are collapsible with client-side persisted state.
> 3. Mobile tap-to-expand row — Phase 5 wallets reveal a trash icon on first tap; Phase 9 holdings reveal P/L% + weight% on first tap (swipe-left still handles Edit/Delete per D-19).

### Accent (10% yellow) reserved-for list — Phase 9 EXACT inventory. No other element may be yellow.

1. **Sticky pill tab "Wallets" — yellow active background** (Phase 3 carry-forward; not redefined here).
2. **`<HoldingSheet>` primary save button** — `Button variant="default"` (yellow background + black text) for the form save action. This IS the primary CTA that commits user work — appropriate yellow use per DESIGN.md L592.
3. **`<AlertDialog>` cancel button** — the non-destructive option uses `Button variant="outline"`, not yellow. The confirm-delete button is red (`Button variant="destructive"`). **No yellow inside the delete confirmation.**

**Justification for minimal yellow inside section content:** Holdings are a dense data surface. The only user-initiating CTA is the save in the Sheet. The `<DashedAddButton>` is NOT yellow (per D-26; matches Phase 5 wallets precedent). Section-level yellow stays reserved for the tab pill indicator and the Sheet save action.

### P/L Semantic Exception — documented deviation from single-yellow rule

> **This is a formal exception recorded in the design contract.** DESIGN.md §Key Characteristics establishes trading-up (#0ecb81) and trading-down (#f6465d) as price-direction semantics applied as **text color**, never as button backgrounds or card surfaces. Phase 4 already uses these tokens for overspent/balance coloring. Phase 9 extends the same pattern to P/L per-holding.

**P/L coloring rules (exact):**

| Condition      | Token                | Hex       | Treatment                                                                                                 | Visual               |
| -------------- | -------------------- | --------- | --------------------------------------------------------------------------------------------------------- | -------------------- |
| P/L > 0 (gain) | `--trading-up`       | `#0ecb81` | Text color on the P/L% value only. Lucide `TrendingUp` 12px same color                                    | `↑ +12.4%` in green  |
| P/L = 0 (flat) | `--muted-foreground` | `#707a8a` | Neutral muted text. No icon.                                                                              | `0.0%` in muted gray |
| P/L < 0 (loss) | `--trading-down`     | `#f6465d` | Text color on the P/L% value only. Lucide `TrendingDown` 12px same color                                  | `↓ −8.2%` in red     |
| Cash (no P/L)  | n/a                  | n/a       | P/L column renders `—` in `--muted-strong`                                                                | `—` placeholder      |
| Delisted       | `--muted-strong`     | `#929aa5` | P/L% shown but entire row dimmed; no gain/loss color override on delisted rows — dimming takes precedence | grayed               |

**Color is never the sole P/L indicator:** The `+`/`−` sign prefix and the `TrendingUp`/`TrendingDown` icon both carry the direction. Screen readers receive the signed percentage value with sign ("+12.4%" reads as "positive 12.4 percent"). No label like "Gain" or "Loss" is appended — the sign + icon is industry-standard and universally understood in financial UIs.

### Destructive (red) reserved-for list — Phase 9 EXACT inventory:

1. **Swipe-reveal Delete button stroke** — `--destructive` text on `--surface-elevated-dark` background. Reveals on left swipe (mobile). Triggers `<HoldingDeleteConfirm>`.
2. **Desktop hover trash icon** — stroke `--destructive`, 28×28 chip, fades in on row hover.
3. **Delete button inside `<HoldingDeleteConfirm>` (AlertDialog)** — `Button variant="destructive"` (shadcn convention).
4. **P/L loss text color** — same `--destructive` / `--trading-down` token. Covered under P/L Semantic Exception above.
5. **`<HoldingSheet>` sheet save button — disabled ring** — 1px `--destructive` ring when server returns an error on save attempt (clears on next interaction).

### Investments section position and structure

The Investments section renders as the **last section** inside the Wallets tab client island, after Reserve wallets. Present only when `investments_enabled = true`. Structure:

```
[Section header: TrendingUp icon + "INVESTMENTS"]
  [Group header: "Broker A" · 42% of portfolio  ▾]            ← collapsible
    [RowDragHandle] [InvestmentRow: AAPL · USD · $4,200 | ↑ +12.4% | 18%]

  [Ungrouped holdings, rendered flat — no group header]
    [RowDragHandle] [InvestmentRow: TSLA · USD · $1,400 | ↑ +5.6%  | 6% ]
  [DashedAddButton: "+ Add investment"]
```

## Price States

Per amendments A2 + D-25: **there are exactly two price states in the UI.** No "pending", no "stale".

### State 1: Normal (price available)

- Row: shows current price converted to buy currency, rendered in `.text-num-md`.
- Sheet (tracked): read-only price display + "Last updated {relative}" hint in `.text-caption --muted-foreground`. No badge.
- Sheet (custom): editable `<AmountInput>`.

### State 2: Delisted

- Trigger: `instruments.active = false` for a linked instrument.
- Row: `opacity-50`, all text `--muted-strong`, inline `Delisted` chip (see Delisted row treatment above).
- Task: `INVESTMENT_INSTRUMENT_DELISTED` emitted (one per affected holding) — task surfaces in Phase 7 Tasks queue, no dedicated banner inside the Investments section.
- Resolution: user archives the holding OR edits it to switch to custom → task resolves.

### Price-blocked on add (rare edge case — A2)

When an on-add instant price fetch fails (rate limit hit / network failure / uncached instrument), the holding save is **blocked**. The form does NOT submit. Instead:

- A `<PriceBlockedBanner>` renders inside the sheet below the current-price field.
- Banner copy (EN): "Couldn't fetch the price — try again in a moment."
- Banner style: background `--surface-elevated-dark`, 1px left border `--destructive` (4px wide, Tailwind `border-l-4`), body text `.text-body-md --body-on-dark`.
- The Save button enters disabled state (`--primary-disabled` bg, `--muted-foreground` text).
- User taps "Retry" text button (`.text-body-md --primary` color, `Button variant="link"`) inline in the banner — re-triggers the fetch. On success, banner clears, save re-enables.
- This state is transient; it does NOT persist a holding with a missing price.

## Group Headers & Collapsible Behavior

Per D-24:

- Each group with ≥1 holding renders a collapsible `<InvestmentGroupHeader>`.
- Collapsed state: shows group name + group-% only; holding rows hidden.
- Expanded state: shows group name + group-% + all holding rows.
- Toggle affordance: full group header row is the tap target (44×44px effective on mobile). `ChevronDown` when expanded, `ChevronRight` when collapsed.
- Persistence: `localStorage` key `inv-group-{budgetId}-{groupSlug}`. Default: expanded.
- Ungrouped holdings render **flat and always visible** — no group header, no collapse affordance.
- Group-% of total: `(group value in budget ccy / total investments value in budget ccy) × 100`, rounded to 1 decimal. Rendered in `.text-num-sm --muted-foreground` trailing the group name.

## Drag-and-Drop Model

Reuses Phase 4+5 `@dnd-kit` patterns with three coexisting gestures on mobile rows:

| Gesture         | Activation                                         | Action                                     |
| --------------- | -------------------------------------------------- | ------------------------------------------ |
| Tap             | Single tap on row body (mobile)                    | Expand/collapse P/L% + weight% second line |
| Swipe-left      | Horizontal swipe ≥ 48px leftward                   | Reveal Edit + Delete action panel          |
| Long-press drag | TouchSensor `{delay: 300, tolerance: 5}` on handle | Activate DnD drag overlay                  |

Desktop coexistence: PointerSensor distance 4px on handle; hover reveals pen + trash.

**DnD semantics:**

| Action                      | Behavior                                                                                      |
| --------------------------- | --------------------------------------------------------------------------------------------- |
| Reorder within group        | Updates `sort_order` within group; optimistic                                                 |
| Reorder ungrouped holdings  | Updates `sort_order` globally; optimistic                                                     |
| Drop holding into a group   | Sets `group` field to group name; optimistic; group-% recalculates                            |
| Drop holding out of a group | Clears `group` field (null); holding renders flat; optimistic                                 |
| Reorder whole groups        | All holdings in group move together; group `sort_order` persisted                             |
| Cross-wallet-section drop   | Rejected server-side; optimistic rollback; toast error "Holdings can only be in Investments." |

**Visual states — identical to Phase 5 DnD table** (source row dims to opacity-0.4, DragOverlay clone at full opacity with 1px `--info-ring` border + `shadow-lg`, valid drop zone tints to `--surface-elevated-dark` at 0.6 opacity with 2px dashed `--info-ring` inset border, invalid drop zone tints to `--destructive` at 0.15 opacity).

### `<HoldingSheet>` lifecycle

1. **Open (add):** `<DashedAddButton>` click → Sheet opens, form empty, search input auto-focused.
2. **Open (edit):** Pen icon / Edit button → Sheet opens, form pre-populated with holding data.
3. **Instrument search:** Typing in name field → after 2s idle OR blur → suggestion dropdown appears. Min 2 characters. Selecting row autofills type + fetches live price (shown as read-only when fetched). "Custom" row in list → clears autofill, all fields manual.
4. **Save (tracked):** Validates all required fields → `clientApiWrite` POST/PATCH → optimistic update → Sheet closes → toast success. If price fetch blocked → `<PriceBlockedBanner>` shown, save blocked.
5. **Save (custom):** Same; current price is whatever user typed; prefills to buy price if user left it untouched.
6. **Dirty close:** User taps Cancel with unsaved changes → `<AlertDialog>` discard-confirm. Confirm discard → sheet closes. Cancel → stays open.
7. **Clean close:** User taps Cancel with no changes → Sheet closes silently (D-18).
8. **Archive from sheet:** Delete button in sheet footer → `<HoldingDeleteConfirm>` opens (separate from sheet — sheet stays behind). Confirm → archive + toast → sheet closes. Cancel → sheet stays open.

### `<InvestmentGroupHeader>` lifecycle

1. Resting expanded: group name + group-% + `ChevronDown`. Full header row is tappable.
2. Tap → `ChevronRight`; holding rows animate collapse (accordion-down/up keyframes from `global.css`).
3. State persisted to `localStorage` immediately on tap.
4. On budget change (different budget opened): reads localStorage for that budgetId; defaults to expanded if no entry.

## Accessibility

- **All drag handles**: `aria-label="Drag to reorder {name}."`, keyboard-activatable via KeyboardSensor (Phase 4+5 carry-forward).
- **`<InvestmentRow>` tap-expand (mobile)**: Row element is `role="button"` + `aria-expanded={true|false}` + `aria-label="Expand {name} details."`. Expanded state changes `aria-expanded`.
- **`<InvestmentGroupHeader>`**: `role="button"` + `aria-expanded={true|false}` + `aria-label="{groupName}, {groupPct}% of portfolio. {Expanded|Collapsed}."`.
- **P/L values**: `aria-label="{+/−}{value}% profit/loss"` so screen readers read "positive 12.4% profit/loss" not just the number.
- **Delisted badge**: rendered as `<span role="status">` so screen readers re-announce when a row transitions to delisted.
- **`<HoldingSheet>`**: Radix Dialog-backed (shadcn `<Sheet>`) — full a11y from Radix.
- **`<AlertDialog>`**: Radix-backed — full a11y.
- **`<PriceBlockedBanner>`**: `role="alert"` so screen readers announce it when it appears.
- **Color is never the sole indicator**: P/L uses `+`/`−` sign prefix AND `TrendingUp`/`TrendingDown` icon in addition to color. Delisted rows are paired with the explicit "Delisted" chip text AND section opacity — never color alone.
- **Section header**: rendered as `<h3>` — screen readers announce section context.
- **Focus order**: section header → group header (collapse toggle) → first row drag handle → row body → swipe-reveal Edit / Delete → next row → `<DashedAddButton>`.
